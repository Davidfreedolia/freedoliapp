# D32 — Decision Engine Architecture

Status: Draft

---

## 1 Core Principle

Freedoliapp must not only present data.

It must generate **operational decisions**.

The system must answer questions such as:

- what should I reorder today
- what shipment should I send to FBA
- which product is at risk of stockout
- where cashflow risk exists

---

## 2 Decision Layer Position

Architecture flow:

Channels  
↓  
Connectors  
↓  
Canonical Data  
↓  
Inventory Ledger  
↓  
Engines  
↓  
Decision Engine  
↓  
Actions / Tasks

---

## 3 Engine Inputs

The decision engine consumes outputs from existing engines:

- Profit Engine
- Reorder Engine
- Cashflow Engine
- Inventory Intelligence
- Margin Compression Alerts

---

## 4 Decision Types

Examples of decisions the system may generate:

- Reorder decision
- Shipment decision
- Stock risk mitigation
- Cashflow warning
- Supplier selection recommendation

---

## 5 Decision Object

Conceptual structure:

decision

Fields:

- decision_id
- org_id
- decision_type
- priority_score
- description
- context_data
- created_at

---

## 6 Priority Scoring

Decisions must be prioritized using signals such as:

- stockout risk
- days of cover
- cash impact
- margin impact
- lead time

The system surfaces the highest priority decisions first.

---

## 7 Output Layer

Decisions appear in:

- Home dashboard
- Operations Planning
- Alerts
- Task queues

---

## 8 Architecture Rules

- decisions must be deterministic
- decisions must be explainable
- decisions must be traceable to source data
- decision logic must remain inside engines

---

## 9 Future Compatibility

The decision layer must support:

- multi-channel commerce
- bundles and assemblies
- supply network routing
- forecasting

## Decision Status

States of a decision.

- open
- acknowledged
- resolved
- dismissed

Purpose:

Allow the system and the user to manage operational actions.

---

## Context Data

context_data contains structured information needed to understand the decision.

Examples:

- variant_id
- recommended_quantity
- supplier_id
- days_of_cover
- expected_stockout_date

The context must always be traceable to canonical system data.

---

## Priority Score

Each decision receives a priority score.

Signals may include:

- stockout risk
- margin impact
- cashflow impact
- lead time
- inventory coverage

The highest priority decisions appear first in the UI.

---

## Decision Lifecycle

Typical lifecycle:

engine detects condition  
↓  
decision created  
↓  
user acknowledges  
↓  
decision resolved or dismissed

## Architecture Guarantees

Decisions must be:

- deterministic
- explainable
- reproducible from system data
- independent from UI logic

## Decision Engine Schema Proposal

Define the conceptual schema needed to persist decisions generated by the system.

---

### decisions

Conceptual table.

Fields:

- id
- org_id
- decision_type
- priority_score
- title
- description
- status
- created_at
- resolved_at

Indexes:

- org_id
- status
- priority_score
- created_at

---

### decision_context

Stores structured context for each decision.

Fields:

- id
- decision_id
- key
- value

Purpose:

Allow decisions to store flexible structured data without modifying schema.

Examples:

- variant_id
- supplier_id
- recommended_quantity
- expected_stockout_date
- days_of_cover

---

### decision_sources

Defines which engine generated the decision.

Fields:

- id
- decision_id
- source_engine
- source_reference

Examples of source_engine:

- reorder_engine
- inventory_engine
- cashflow_engine
- profit_engine

---

### decision_events

Optional lifecycle log.

Fields:

- id
- decision_id
- event_type
- event_data
- created_at

Examples of event_type:

- created
- acknowledged
- resolved
- dismissed

---

### Architecture rules

- decisions must be append-safe
- decisions must be traceable to engines
- decisions must remain reproducible from system data
